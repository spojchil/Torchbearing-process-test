# 安全策略

Torchbearing 会接触 Grafana、数据源、监控指标和外部服务配置。安全问题按照最小暴露原则处理。

## 支持范围

项目当前只维护正式仓库 `main` 上的最新版本。个人 Fork、未合并分支和历史提交不单独提供安全修复承诺。

## 报告安全问题

发现以下问题时，不要在普通 Issue、PR 评论或公开讨论中粘贴完整细节：

- Token、密码、Cookie、API Key、私钥或其他凭据泄漏；
- 未授权访问、越权查询或未经确认的写操作；
- 敏感指标、日志、用户信息或内部地址泄漏；
- 命令执行、请求伪造、路径穿越或注入；
- 依赖、构建链或 GitHub Actions 供应链风险。

请通过团队约定的私密渠道联系仓库创建者或仓库协作者，并提供：

1. 受影响的版本或 commit；
2. 问题类型和可能影响；
3. 最小复现步骤；
4. 已采取的临时措施；
5. 脱敏后的日志或截图。

维护者负责确认报告、限制扩散并协调修复。在完成风险评估前，不公开复现细节。

## 凭据与数据

- `.env`、Token、密码、Cookie 和私钥不得提交到 Git；
- 示例配置只使用无敏感性的默认值或占位符；
- 测试优先使用 Mock、临时凭据和脱敏数据；
- 日志、截图和验证输出在提交前检查敏感信息；
- 向 AI 服务提供上下文前移除凭据、内部地址和真实用户数据；
- 凭据一旦泄漏，立即撤销或轮换，不以删除 Git 文件代替轮换。

## Fork 与自动化

Fork PR 的自动化工作流不得获得仓库 Secrets 或写 Token。自动化尚未建立时，同样不得把凭据写入本地验证记录、PR 正文或截图。

工作流、依赖更新和安全修复均通过独立 PR、必要验证和人工 Review，不自动合并破坏性变更。
